# 待解决问题

更新日期：2026-07-13

## 高优先级

### 1. 旧 Memory 项目目录不会自动迁移

- 路径规范化算法修改后，`D:\Code\AxiomWeave` 和 `D:\code\AxiomWeave` 会统一映射到新目录 `dfc174bd87dff0ec`。
- 旧算法生成的 `ed72c87b157c9f92` 与 `afed45de5f28812c` 不会被自动读取或合并。
- 当前两个旧目录均为空，可以在关闭 AxiomWeave 后手动删除；如果以后遇到包含记忆文件的旧目录，必须先合并和备份，不能直接删除。
- 后续可增加一次性迁移命令，扫描 Session 元数据中的旧工作目录并迁移 Memory。

### 2. 核心模块自动化测试不足

- 仍需覆盖 Agent Loop、权限矩阵、Session 恢复、Memory CRUD 与召回、Skills 演化和 MCP 异常处理。
- 当前新增测试只锁定 Windows 路径大小写问题，不能代表整个 Runtime 已具备生产稳定性。

### 3. 权限系统仍是实验性保护

- 需要完善基于能力的权限模型、危险命令识别和子 Agent 权限继承。
- 当前权限检查不能替代操作系统级沙箱或容器隔离。

## 中优先级

### 4. Memory 保存入口不够明确

- `/memory` 只能查看已有长期记忆，没有 `/memory-create` 等直接创建命令。
- 长期记忆目前依赖模型按 System Prompt 调用 `write_file`，用户容易误以为 `--resume` 会自动生成 Memory。

### 5. MCP 生命周期与异常恢复需要加强

- 需要补充连接超时、子进程退出、请求取消和重连策略。

### 6. 工程质量基础设施待补齐

- 尚未接入 CI、类型检查、代码规范、覆盖率报告和可重复运行的 Benchmark/Eval。
- Provider、Runtime、Context 和 Permission 的模块边界仍可继续拆分。
