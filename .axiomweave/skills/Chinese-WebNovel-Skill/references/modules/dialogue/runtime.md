# 对白模块运行规则

## 一、什么时候必须调用

出现这些信号时，优先调用本模块：

- 用户要你补一段高张力对白
- 文本里的对白像解释，不像冲突
- 人物说话像同一个人
- 关系戏到关键处，只剩互相喊话
- 本来该在对白里爆出来的信息，被作者旁白替代了

## 二、先判断问题卡在哪一层

### 第 1 层：空对白

典型表现：

- 套话多
- 情绪词多
- 信息量低

优先补：

- 具体刀口
- 现实利益
- 新信息

### 第 2 层：说明书对白

典型表现：

- 人物一边吵架一边讲背景
- 一句台词里塞三层解释

优先补：

- 把设定拆回动作或前文
- 把对白改成压迫与试探

### 第 3 层：有人设，没有声音

典型表现：

- 理论上人物不同，但台词风格一样
- 谁说都像作者在说

优先补：

- 身份感
- 教养感
- 情绪习惯

### 第 4 层：有火花，但不推进局面

典型表现：

- 吵得热闹
- 关系、利益、决定都没变化

优先补：

- 让一句话落成判决、威胁、逼问或站队

## 三、例库映射

### 1. 如果问题是“地位和家规施压不够狠”

先看：

- 正例：`good_examples.md` A 组
- 反例：`bad_examples.md` A 组

### 2. 如果问题是“情感绑架写得太直白”

先看：

- 正例：`good_examples.md` B 组
- 反例：`bad_examples.md` B 组

### 3. 如果问题是“需要一句摊牌或判决”

先看：

- 正例：`good_examples.md` C 组
- 反例：`bad_examples.md` C 组

### 4. 如果问题是“人物声音不分”

先看：

- 正例：`good_examples.md` A 组、C 组
- 反例：`bad_examples.md` D 组

## 四、动笔前先写对白压力计划

正文前先明确：

- 谁要压谁：
- 压力靠身份、感情还是利益：
- 哪句是刀口：
- 哪句说完以后，局面必须变化：

## 五、任务类型处理

### 1. 如果用户要你诊断对白

先输出：

- 空在什么地方
- 压力从哪一层缺了
- 这段更像说明书、空话还是失真

### 2. 如果用户要你改写对白

先做：

- 确定双方当前目标
- 确定这轮对话必须爆出的信息
- 确定谁占上风

### 3. 如果用户要你新写对白

默认顺序：

1. 先定关系
2. 再定利益
3. 再定刀口
4. 最后写词面

## 六、专项硬规则

- 不要用套话顶替冲突。
- 不要让人物一边吵一边讲百科。
- 不要每句都很长。
- 不要只喊情绪，不落刀口。
- 不要让所有角色说话都像作者。
