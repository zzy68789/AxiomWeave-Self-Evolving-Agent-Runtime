# 前置规划模块运行规则

## 一、什么时候必须调用

出现这些信号时，优先调用本模块：

- 用户明确在问“这个点子能不能写”
- 用户要你做选材、构思、hook、premise、故事引擎或长度判断
- 简介里有设定，但第一章会发生什么不清
- 看上去能写一章，但看不出怎么跑过前 10-20 章
- 还没开写正文，就已经感觉卖点发虚

## 二、先判断卡在哪一层

### 第 1 层：题材 / 消费点层

典型表现：

- 题材很混
- 读者到底来吃什么说不清

优先补：

- 题材归属
- 主要消费点
- 主卖线

### 第 2 层：卖点 / 异常局面层

典型表现：

- 一句话讲不清最抓人的点
- hook 像背景介绍，不像成交句

优先补：

- 异常局面
- 身份反差
- 危机或诱因

### 第 3 层：主角处境 / 第一麻烦层

典型表现：

- 主角很特别，但没有被逼着动
- 第一章到第三章看不出眼前任务

优先补：

- 当前麻烦
- 不行动的损失
- 第一波冲突

### 第 4 层：故事引擎层

典型表现：

- 能写一句 hook，但十章后没路
- 点子像一次性反转，不像连载结构

优先补：

- 持续推进链
- 升级或调查路径
- 每一步的局面变化

### 第 5 层：长度 / 第一卷层

典型表现：

- 点子太薄，却想硬拉长篇
- 点子其实能连载，却没有卷一目标

优先补：

- 目标长度
- 第一卷最小目标
- 卷末变化

## 四、动笔前先写最小骨架

- 这是一本什么题材的网文：
- 读者主要来吃什么：
- 主角眼前最紧迫的问题是什么：
- 接下来 10-20 章靠什么持续推进：
- 更适合什么长度：
- 第一卷最小目标和卷末变化是什么：

## 五、任务类型处理

### 1. 如果用户要你诊断一个点子值不值得写

先输出：

- 卡在题材、卖点、处境、引擎还是长度
- 最缺的是哪一句
- 要成立，最先该补什么

### 2. 如果用户要你重写 hook 或 premise

先做：

- 不急着润色句子
- 先补骨架，再压成一句 hook 或一段 premise

### 3. 如果用户要你直接做规划

默认先过本模块，再把下游问题分给：

- `opening`：处理开头成交
- `volume_outline`：处理黄金三章和章纲

## 六、专项硬规则

- 不要用大设定代替眼前麻烦。
- 不要用主题词代替故事引擎。
- 不要把最强卖点埋在一长段介绍后面。
- 不要把一次性点子硬推荐成长篇连载。
- 最小骨架没立住前，不要急着开正文。
