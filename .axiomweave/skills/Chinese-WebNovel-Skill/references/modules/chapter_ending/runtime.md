# 章末模块运行规则

## 一、什么时候必须调用

出现这些信号时，优先调用本模块：

- 用户明确要求补章末、补追更点
- 文本收在总结、讲理、抒情上
- 章末没把本章变化收住
- 章末虽然炸，但下一章接不上
- 轻喜文收尾太平，关系戏收尾太散

## 二、先判断问题卡在哪一层

### 第 1 层：没有落点

典型表现：

- 写完就停
- 最后一段像散场

优先补：

- 最后一刀落点
- 本章最值钱的变化

### 第 2 层：有落点，没有后劲

典型表现：

- 收尾完整
- 但没有追更拉力

优先补：

- 未尽之力
- 下一章自然入口

### 第 3 层：硬钩感太重

典型表现：

- 每章都像电视剧广告点
- 钩子不是从本章生长出来的

优先补：

- 从关系、情绪、决定里长出钩子

### 第 4 层：上一章和下一章脱节

典型表现：

- 章末明明蓄力了
- 下一场第一拍没接住

优先补：

- 和转场模块联动
- 让下一章第一拍兑现或反打上一章余力

## 三、例库映射

### 1. 如果问题是“想用情绪余韵收”

先看：

- 正例：`good_examples.md` A 组
- 反例：`bad_examples.md` A 组

### 2. 如果问题是“轻喜文收尾太平”

先看：

- 正例：`good_examples.md` B 组
- 反例：`bad_examples.md` B 组

### 3. 如果问题是“想收在决定或局面变化上”

先看：

- 正例：`good_examples.md` C 组
- 反例：`bad_examples.md` C 组

### 4. 如果问题是“章末和下一场接不上”

先看：

- 正例：`good_examples.md` D 组
- 反例：`bad_examples.md` D 组
- 再联动：`transition/runtime.md`

## 四、动笔前先写章末计划

正文前先明确：

- 本章最后一拍收什么：
- 收在谁的动作、谁的话还是谁的决定上：
- 读者下一秒最想知道什么：
- 下一章第一拍怎么接：

## 五、任务类型处理

### 1. 如果用户要你诊断章末

先输出：

- 现在收在什么上
- 为什么不够有力
- 是该补余韵、反打、决定还是承接

### 2. 如果用户要你重写章末

先做：

- 找出本章最值钱的变化
- 判断最适合的落点类型
- 明确下一章第一拍

### 3. 如果用户要你直接写章末

默认顺序：

1. 定本章变化
2. 定落点
3. 定余力
4. 最后落词

## 六、专项硬规则

- 不要收在总结句。
- 不要收在讲道理。
- 不要把悬念解释没。
- 不要为了钩而钩。
- 不要忘了下一章还得接。
